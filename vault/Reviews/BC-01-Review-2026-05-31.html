⛔ Critical Issues — Block Merge

C-1  ·  RS256 Required — HS256 Implemented (Invariant #12 Violation)

Spec §6.2 inv #12 Spec §9.2 JwtSigner port JwtSigner.cs

Required Access tokens must be signed with RS256 (asymmetric, private-key signing). The spec states this explicitly in invariant #12, in the port definition for JwtSigner, and in the acceptance criteria for US-3.4.3-04.

Actual JwtSigner.cs uses SecurityAlgorithms.HmacSha256 — a symmetric HMAC-SHA256 algorithm — with a shared secret read from configuration (JwtSettings.Secret).

Impact Any process that has access to the shared secret can forge valid tokens. All services that call TokenValidationApi.Validate() would be vulnerable. The public contract stability guarantee (referenced in BC-2, BC-3 packages) is broken. This is the most severe security defect in the PR.

Fix: Replace the HmacSha256 key with an RSA key pair. Load the private key (PEM/PKCS8) from appsettings.Development.json for dev; use a key vault reference for production. Sign with SecurityAlgorithms.RsaSha256 and validate with the public key only.

C-2  ·  PasswordPolicyService.Validate() Always Returns Success (Invariant #4 Violation)

Spec §7.1 Spec §6.2 inv #4 PasswordPolicyService.cs

Required PasswordPolicyService.Validate(candidate) is a pure domain service that must enforce: min length 10, ≥3 character classes, no trivial sequences. Returns Error("E-REG-INVALID-PASSWORD", …) on failure.

Actual The method body contains TODO comments and unconditionally returns Result.Success. The FluentValidation validator on ProvisionCredentialCommand provides a partial check at the command level, but the domain service — which is also called on the password-change and password-reset paths — is inert.

Impact A password with fewer than 10 characters or a single character class can be set via the change-password or complete-reset handlers, which bypass the command validator and rely on the domain service. Domain invariant #4 is violated.

Fix: Implement the body: check candidate.Value.Length >= 10, count character-class presence (lowercase, uppercase, digit, symbol), detect trivial sequences (e.g. "aaaaaaaaaa", "1234567890"). Map to E-REG-INVALID-PASSWORD or the caller-supplied prefix (E-RESET-INVALID-PASSWORD on the reset path).

C-3  ·  No Outbox / Inbox Integration — Integration Events Not Durably Persisted

Spec §8.1 Plan Phase 8.1 IdentityAccessDbContext.cs

Required IdentityAccessDbContext must implement IOutboxInboxDbContext and expose DbSet<OutboxMessage> + DbSet<InboxMessage>. Integration events (UserRegisteredIntegrationEvent, UserAccountSuspendedIntegrationEvent, etc.) must be written into the outbox table in the same transaction as the aggregate change.

Actual IdentityAccessDbContext : DbContext — no IOutboxInboxDbContext, no outbox/inbox DbSets. Integration events raised via RaiseDomainEvent() are dispatched in-process by the PublishDomainEventsInterceptor (direct mode), with no durable persistence.

Impact If the process crashes between the SaveChanges returning and the in-process MediatR publish completing, integration events are silently dropped. At-least-once delivery is not guaranteed. BC-9 (Notification), BC-10 (Reporting), and BC-12 (Content Management) consume these events and will miss events during crashes or restarts.

Fix: Implement IOutboxInboxDbContext on the DbContext. Configure OutboxMessage and InboxMessage entities in OnModelCreating under schema identity_access. Ensure the PublishDomainEventsInterceptor writes integration events to the outbox table instead of publishing directly.

C-4  ·  Consumed Integration Event Handlers Not Implemented (§9.1 Gap)

Spec §9.1 Plan Phase 13

Required Two inbound integration event handlers with inbox idempotency:

  • IdentityVerifiedByGovernmentIntegrationEvent → load account, call account.ApplyGovernmentIdentityVerified(registry), save. Idempotent (inbox check).
  • IdentityVerificationFailedIntegrationEvent → record failure for audit. No status change. Idempotent.

Actual No handler classes found in the Application or Infrastructure layers. The events are defined in IntegrationEvents.cs in the Contracts project but nothing reacts to them within this module.

Impact When BC-8 publishes IdentityVerifiedByGovernmentIntegrationEvent, the UserAccount.IdentityVerified flag will never be set. The AdminUserDetailDto will always show IdentityVerified = false regardless of actual verification status. Acceptance criteria for government-identity verification cannot pass.

Fix: Add IdentityVerifiedByGovernmentEventHandler : INotificationHandler<IdentityVerifiedByGovernmentIntegrationEvent> and IdentityVerificationFailedEventHandler in Application/IntegrationEvents/. Each handler must check inbox_messages before processing (idempotency per Shared Foundations §6.3).

⚠️ Moderate Issues — Should Fix Before Merge

M-1  ·  Credential VO — Nullable Parameters Not Updated (Plan §1.3)

Plan §1.3 The implementation plan notes: "Minor update to Credential.Create: Change nullable params to non-nullable for cleaner design." This change should be confirmed; if the factory still accepts nullable arguments, callers can pass null through without a compile-time error, relying entirely on runtime checks.

M-2  ·  IUserAccountRepository Missing Explicit Update Method

Spec §11.3 The repository contract in §11.3 lists Update and a UnitOfWork (SaveChanges). The interface exposes SaveChangesAsync but no UpdateAsync. While EF Core change-tracking handles this implicitly for tracked entities, the interface contract should match the spec so that future non-tracked scenarios (e.g. projection repositories) work correctly.

M-3  ·  PasswordHistory SHA-256 Verifier — Confirm Keeps Last 3 Only

Plan Appendix The design issue around Argon2id + PasswordHistory is well-documented and the SHA-256 verifier approach is correct. However, the implementation must be confirmed to: (1) use SHA256.HashData on the raw password before any Argon2id hashing, (2) cap the list at 3 entries by removing the oldest on overflow, and (3) store only the SHA-256 hex string, never the raw password. Without tests (see C-5), this cannot be verified.

M-4  ·  AdminActionLog Not Verified in Same Unit of Work

Spec §6.4, Invariant #11 Every admin command (Suspend, Reinstate, Deactivate, Unlock, PasswordResetIssued, RoleAssigned, Viewed, ApprovedEmployer, RejectedEmployer) must write an AdminActionLog row in the same unit of work as the aggregate change. Since no application handler tests exist (C-5), this atomicity guarantee is currently unverified. A crash between the aggregate save and the log insert would produce a silent audit gap.

M-5  ·  IssueOAuthTokenCommand — PKCE Validation Completeness

Spec §10.1, US-3.4.3-04 The OAuth 2.0 Authorization-Code flow with PKCE capability is listed as in-scope. The command and handler exist, but the PKCE code_verifier / code_challenge validation logic requires explicit review. Without this, the PKCE protection against authorization-code interception is absent, undermining the security model for third-party portal integrations.

ℹ️ Minor Issues — Acceptable Tech Debt (Post-Merge)

N-1  ·  Stub Adapters: BreachCheck, OtpDelivery, RateLimiter

All three stubs are explicitly approved by the spec for the dev/exercise phase. BreachCheckPortStub uses an in-memory blocklist, OtpDeliveryPortStub logs the plaintext code, and RateLimiterPortStub uses an in-process ConcurrentDictionary. Port shapes match spec exactly — production adapters drop in without changing application code. Acceptable, documented.

N-2  ·  In-Memory Database — No EF Migrations

The spec explicitly permits starting with UseInMemoryDatabase. The implementation plan documents the one-line switch to UseNpgsql/UseSqlServer when a real DB is needed. Acceptable as a first iteration.

N-3  ·  RateLimiterStub Does Not Survive Restarts

The in-memory sliding window resets on process restart. Acceptable for dev; document as a known limitation and flag for Redis replacement before production deployment.

N-4  ·  BreachCheck Blocklist Is Minimal

The stub likely covers only a handful of known-bad passwords. Not a security risk in dev, but the production adapter should implement the k-anonymity range query against HaveIBeenPwned before any real user traffic.

N-5  ·  No OtpIssued Handler Wired to OtpDeliveryPort

Plan Phase 3.5 / 5 The spec says the internal OtpIssued domain event should trigger a call to OtpDeliveryPort.Send() inside the ProvisionCredentialCommandHandler (or a dedicated domain event handler). This wiring needs to be confirmed — if the handler calls the port directly after OtpChallenge.Issue(), it is fine. If it relies on a missing INotificationHandler<OtpIssuedEvent>, the delivery call will be silently absent.

N-6  ·  PR Scope Mixing — .NET 10 Upgrade + Feature Work

The branch mixes two unrelated concerns: the .NET 9 → .NET 10 / C# 14 upgrade (build fixes, JwtBearer reference updates) and the full BC-01 feature implementation. For traceability and bisect-ability, these should be in separate PRs. Not a functional blocker, but worth noting for future practice.
